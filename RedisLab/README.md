# RedisLab — cache-aside, stampede и инвалидация

Pet-проект №13 из плана. Закрывает раздел 11 из «200 ответов»: **Кэширование и Redis**.

## Предметная область

Redis 7 в Docker + «медленная БД» (200 мс на запрос, со счётчиком обращений).
Три сценария доказывают каждое утверждение числом реальных походов в БД.

## Как запустить

```bash
docker compose up -d
dotnet run --project src/RedisLab.App -c Release   # сценарии идемпотентны
```

## Результаты с этой машины

| Сценарий | Результат |
|---|---|
| **Cache-aside** | miss: 225 мс (поход в БД) → hit: ~1 мс из Redis |
| **Stampede**, 50 конкурентных промахов | наивный: **50** походов в БД · single-flight (SemaphoreSlim): **1** · распределённый лок (SET NX): **1** |
| **Инвалидация** | обновление БД + `DEL` ключа → следующий читатель наполняет кэш свежим; TTL — страховка |

## Что где смотреть (`src/RedisLab.App/Scenarios.cs`)

- **Cache-aside** — базовый паттерн; `SET` с TTL одним вызовом (не SET+EXPIRE — между
  двумя командами можно упасть и получить ключ-бессмертник).
- **Single-flight** — `SemaphoreSlim` на ключ + double-check после захвата. Решает
  stampede внутри одного процесса.
- **Распределённый лок** — `SET NX PX` (атомарно «поставь, если нет, с автопротуханием»),
  токен владельца + снятие через Lua compare-and-delete: нельзя удалить чужой лок,
  если свой протух. TTL на локе обязателен — упавший держатель не заблокирует ключ навечно.
- **Инвалидация** — `DEL` при записи, а не перезапись: у write-through есть гонка
  «две записи обновили кэш в разном порядке».
