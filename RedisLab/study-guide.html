2Ключевые идеи

Cache-aside vs read-through/write-through. В cache-aside кэш пассивен: приложение само читает, само наполняет, само инвалидирует. Read-through — наполнение прячется в кэш-прослойку; write-through — запись идёт сквозь кэш в БД. Cache-aside — дефолт индустрии: простой, кэш можно снести целиком без потери данных.

Stampede (dog-piling). Горячий ключ протух → N конкурентных промахов → N одинаковых запросов в БД. В сценарии: 50 походов вместо 1. Смертельно на дорогих запросах: БД складывается ровно тогда, когда кэш «должен спасать». Три защиты:

  • Single-flight в процессе — SemaphoreSlim на ключ + double-check после захвата (пока ждал — сосед уже наполнил кэш);
  • Распределённый лок — SET NX PX: работает между всеми инстансами сервиса;
  • Early refresh — обновлять значение фоном ДО истечения TTL: промахов не бывает вовсе (ценой фоновой работы).

Анатомия распределённого лока (всё в GetWithDistributedLockAsync):

SET lock:key {случайный-токен} NX PX 5000   -- атомарно: поставь если нет, с TTL
...работа...
-- снятие: ТОЛЬКО свой лок, атомарно через Lua (compare-and-delete)
if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) end

Три обязательных детали: TTL (упавший держатель не вечен), токен владельца, Lua-снятие (наш лок мог протухнуть, и ключ уже держит другой — нельзя удалить чужой). Это мини-Redlock на одном узле — и честный ответ, что для мультиузловой строгости нужен Redlock/ZooKeeper, но чаще хватает этого.

Инвалидация: DEL, не перезапись. «Обновил БД → DEL ключа» проще и безопаснее write-through: у перезаписи есть гонка (две записи обновили кэш в противоположном порядке — в кэше старое навсегда). TTL остаётся страховкой. Полной согласованности кэша с БД не существует — выбираем допустимое окно рассинхрона.

Классика для собеса — почему SET+EXPIRE двумя командами это баг: упади процесс между ними — ключ останется без TTL навсегда. Всё, что должно быть атомарным в Redis, делается одной командой, MULTI/EXEC или Lua.

3Мостик к вопросам собеса

Паттерны: cache-aside / read-through / write-through / write-behind; stampede и все три защиты; инвалидация и её гонки; выбор TTL; какие данные кэшировать нельзя.
Redis: почему быстрый (память + один поток на команды = нет локов); структуры (string, hash, list, set, zset, stream) и типичные применения; distributed lock и Redlock; политики eviction (allkeys-lru vs volatile-*); persistence (RDB vs AOF); ConnectionMultiplexer.
Связка с другими проектами: in-memory кэш TtlCache — первый уровень; Redis — второй, разделяемый между инстансами (и переживает деплой). HybridCache в .NET 9 объединяет оба уровня + защиту от stampede из коробки.
