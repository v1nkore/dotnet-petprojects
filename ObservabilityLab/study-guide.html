2Ключевые идеи

Три сигнала. Метрики — агрегаты, дёшевы, отвечают «что происходит» (алерты). Трейсы — путь конкретного запроса, отвечают «где именно тормозит/падает». Логи — детали в точке. Связка: алерт по метрике → трейс медленного запроса → логи по trace id.

Activity = span. В .NET трейсинг встроен в BCL: ActivitySource.StartActivity() создаёт спан, OTel SDK подписывается на источники по имени и экспортирует. ASP.NET Core, HttpClient, Npgsql уже инструментированы — свои спаны нужны только для того, чего SDK не видит (Kafka).

Propagation. Контекст (trace id + parent span id) едет между процессами в заголовке traceparent. HTTP — автоматически. Kafka, очереди, фоновые джобы — руками: Inject/Extract через Propagators.DefaultTextMapPropagator. Правило: любой асинхронный разрыв требует явного проноса контекста.

RED поверх готовых метрик. Rate = rate(..._count[1m]), Errors = то же с фильтром по 5xx, Duration = histogram_quantile по бакетам. Ничего специально писать не пришлось — инструментация ASP.NET Core уже отдаёт гистограмму длительностей по роутам.

Главная находка — трейс рвётся на Outbox. Первая версия: 2 спана вместо 21. Публикатор живёт в фоновом цикле — Activity от HTTP-запроса давно нет, publish начинал новый трейс. Решение: сохранить Activity.Current?.Id (W3C traceparent) в outbox-строку при записи и передать как родителя при публикации. Этот паттерн спрашивают у тех, кто заявляет «настраивал трейсинг в микросервисах»: связал ли ты асинхронные куски.
Бонус, который сделал трейс золотым: HttpClient-инструментация показывает КАЖДЫЙ ретрай Polly отдельным спаном — в Jaeger видно три красных POST /charge (500) и четвёртый зелёный. Идеальная картинка «как наблюдаемость объясняет поведение resilience-политик».

3Мостик к вопросам собеса

Раздел 15: логи/метрики/трейсы и их назначение; correlation id vs trace id; как работает propagation (HTTP и очереди); спан, его атрибуты, kind (Producer/Consumer/Server/Client); сэмплирование (head vs tail) и зачем; RED/USE; pull vs push (Prometheus vs OTLP push); structured logging и связь логов с trace id; во что стоит алертить (симптомы, не причины).
