2Ключевые идеи

Декларативность. Ты не приказываешь «запусти контейнер», а описываешь желаемое состояние (2 реплики, такой образ, такие пробы) — контроллеры непрерывно доводят реальность до декларации. Умер под — поднимется сам; это не фича деплоя, а свойство модели.

Иерархия объектов: Pod (эфемерный, свой IP) ← ReplicaSet (держит N копий) ← Deployment (управляет ReplicaSet'ами при обновлениях — поэтому откат это «переключиться на старый ReplicaSet», мгновенно). Service — стабильное имя + балансировка над меняющимися подами.

Rolling update, увиденный руками: новый под с битым образом завис в ErrImageNeverPull, а старые реплики продолжали отвечать — k8s не убивает работающее, пока новое не пройдёт readiness. maxSurge/maxUnavailable управляют темпом.

Helm = пакетный менеджер + шаблонизатор + история релизов. Чарт — манифесты с дырками под values; релиз — установленный чарт с ревизиями (helm rollback). Жемчужина чарта: checksum/config — хэш ConfigMap'а в аннотации пода, чтобы смена конфига перекатывала поды (иначе живут со старым до случайного рестарта).

CI/CD-цепочка: тесты гейтят сборку образа, образ тегируется SHA коммита, деплой через helm upgrade --wait — если поды не прошли probes, шаг падает и есть что откатывать. Environment protection — ручное подтверждение на прод.

Диагностика OOMKilled (частый вопрос): kubectl describe pod → Last State: Terminated, Reason: OOMKilled, exit code 137. Причины: лимит ниже реального потребления, утечка, или .NET до контейнерных лимитов не знал о cgroup (сейчас знает: Server GC считает память от лимита). Лечение: поднять limits / чинить утечку (см. PerfLab) / проверить DOTNET_GCHeapHardLimit.
