1Предметная область

Домен взыскания — твой рабочий контекст: Debtor → Debt → Payment. 100 должников, 300 долгов, 300 000 платежей в настоящем Postgres 16 (docker-compose, порт 5455). Консольное приложение прогоняет 4 сценария и каждое утверждение доказывает цифрами: числом SQL-запросов, миллисекундами, строками плана.

Почему это важно: ORM — самый частый источник проблем производительности в проде, и все они сводятся к трём вопросам: сколько запросов ушло (N+1), что тащим и трекаем зря (change tracking), как база исполняет запрос (план). Плюс конкурентные правки — четвёртый всадник.

3N+1: три версии одного чтения

СпособЗапросовВремяЧто не так / что так
lazy loading101306 мсОбращение к d.Debts выглядит как чтение свойства, а прокси молча ходит в базу. Опаснее всего то, что код работает — просто в 100 раз дороже.

Правило: для чтения — проекция в DTO; Include — когда нужна именно сущность целиком (для последующего изменения); lazy loading в бэкенде — почти всегда выключить. Отдельно запомни: Include с двумя-тремя коллекциями порождает cartesian explosion — лечится AsSplitQuery().

4Change tracking: цена по умолчанию

Каждая затреканная сущность — это снапшот всех значений (для последующего diff в SaveChanges) плюс запись в identity map. На 50 000 строк: 313 мс с трекингом против 53 мс без — в 6 раз. Identity map — полезный побочный эффект: два запроса одной строки в одном контексте вернут один и тот же объект (сценарий это печатает: ReferenceEquals = True).

Ловушка, пойманная этим же проектом: присваивание того же значения — не изменение. SaveChanges сравнивает снапшоты и, если diff пуст, вообще не отправляет UPDATE. Из-за этого сценарий конкуренции «ломался» при повторном запуске: статус уже был Restructured, UPDATE не уходил, xmin не менялся — и конфликт не возникал. Понимание «SaveChanges — это diff снапшотов» объясняет кучу «мистики» EF.

5Optimistic concurrency через xmin

Сценарий «два оператора открыли одну карточку долга»: оба читают строку, оба сохраняют. Без токена второй молча затирает первого — классический lost update. С токеном EF генерирует:

UPDATE "Debts" SET ... WHERE "Id" = @id AND xmin = @версия_на_момент_чтения

0 затронутых строк → DbUpdateConcurrencyException. Дальше — осознанное решение: перечитать и повторить (в сценарии), смержить по полям или отдать пользователю 409.

Фишка Postgres: в SQL Server для этого нужна колонка rowversion и миграция. В Postgres токен бесплатный — системная колонка xmin (id транзакции, менявшей строку последней) уже есть у каждой строки: uint Version + IsRowVersion() у Npgsql, и никакой миграции. Заодно это дверь в разговор про MVCC: версии строк, зачем VACUUM, откуда bloat.

Запрос-отчёт «суммы подтверждённых платежей за июнь по долгам» на 300k строк:

Ключевые строки, которые надо уметь читать: узел скана (Seq/Index/Index Only/Bitmap), Rows Removed by Filter (сколько прочитано впустую), Buffers (реальное I/O), расхождение rows= оценки и факта (кривая статистика → ANALYZE).

Анатомия индекса ("Status", "PaidAt") INCLUDE ("DebtId", "Amount"):

  • порядок колонок: сначала равенство (Status = 1), потом диапазон (PaidAt BETWEEN) — наоборот индекс почти бесполезен, диапазон «разрезает» дерево;
  • INCLUDE кладёт колонки в листья индекса, не включая их в ключ — запрос покрыт целиком, к heap не ходим (Index Only Scan);
  • но Index Only Scan случится, только если visibility map актуальна — см. нюанс ниже.

7Три нюанса, пойманных в этом проекте

1. Index Only Scan не появился после CREATE INDEX + ANALYZE. Планировщик выбирал Bitmap Heap Scan. Причина — MVCC: индекс не хранит информацию о видимости версий строк, и без актуальной visibility map Postgres обязан заглядывать в heap для каждой строки. VACUUM ANALYZE — и Index Only Scan появился (12.1 → 2.4 мс). На собесе это выделяет из толпы: большинство знает «индекс ускоряет», единицы — почему покрывающий индекс может не сработать.

2. UPDATE, который не отправился. Присваивание того же значения не меняет снапшот → SaveChanges не шлёт UPDATE → xmin не растёт. См. раздел 4.

3. AddRange на 300k строк — это боль. Каждая сущность трекается, вставки построчные. generate_series одним оператором — доли секунды. В реале: EF для OLTP, для bulk — COPY / bulk-библиотеки / raw SQL.
