PetProjects — навигация

Каждый проект закрывает разделы из «200 ответов» и пункт чек-листа. Порядок карточек = порядок разделов в файле вопросов: изучаешь раздел → разбираешь проект → возвращаешься к вопросам.
11 кодовых + 3 письменных модуля
17 из 17 разделов «200 ответов» закрыто
~1–3 вечера на разбор каждого
Как разбирать: открой study-guide проекта (кнопка фиолетовая) — там маршрут по коду, «почему так написано», языковые фишки и пойманные баги. README — краткая выжимка с реальными цифрами и командами запуска. После разбора — конспект «что понял» в 5–10 строк.

Готовые проекты

1 · TtlCache

готов
Разделы 1–2 · C# runtime · Async и многопоточность

Потокобезопасный in-memory кэш с TTL: ленивое протухание при чтении + фоновый свипер на PeriodicTimer. Время — инжектируемая зависимость (TimeProvider), поэтому 11 тестов на TTL пробегают за 142 мс без единого Task.Delay. Плюс бенчмарк трёх стратегий синхронизации.

Бенчмарк: ConcurrentDictionary быстрее lock в 64× на 8 потоках (3 мс против 193 мс, 1571 lock contention) — и, сюрприз, быстрее даже на одном потоке. ReaderWriterLockSlim — посередине и почему это ловушка.

Study-guide README
Запуск: dotnet test · демо dotnet run --project src/TtlCache.Demo -c Release · без инфраструктуры

2 · AspNetLab

готов
Раздел 3 · ASP.NET Core

Сервис нотификаций как каркас для «внутренностей» ASP.NET Core: свои middleware (correlation id + тайминги), кастомный action-фильтр, воркер на BackgroundService + bounded Channel, Options с валидацией на старте, ProblemDetails. Ловушка captive dependency включается флагом и роняет приложение на старте — так и задумано.

Два пойманных бага — готовые истории на собес: [property: Required] на позиционном record (500 в рантайме) и enum-сериализация как часть контракта API.

Study-guide README
Запуск: dotnet test · dotnet run --project src/AspNetLab.Api · ловушка: -- --BreakDi=true

3 · EfCoreLab

готов
Разделы 4–5 · EF Core · Базы данных и SQL

Настоящий Postgres 16 и домен взыскания (Debtor → Debt → Payment, 300k строк). Четыре сценария с доказательствами цифрами: N+1 (101 запрос → 1, считает DbCommandInterceptor), цена change tracking (6×), optimistic concurrency через xmin, EXPLAIN ANALYZE до/после покрывающего индекса.

Seq Scan 12.1 мс → Index Only Scan 2.4 мс — и нюанс, что без VACUUM (visibility map) Index Only Scan не появится вовсе. Плюс «UPDATE, который не отправился» и почему EF — не для bulk-вставок.

Study-guide README
Запуск: docker compose up -d → dotnet run --project src/EfCoreLab.App -c Release · сценарии идемпотентны

4 · OrderFlow

готов
Разделы 8–9 · Микросервисы · Kafka

Флагман: Orders + Payments + намеренно нестабильный «банк» (35% пятисоток, зависания, лимиты). Kafka (KRaft), Outbox одной транзакцией с бизнес-данными, идемпотентные консьюмеры, saga-хореография с компенсацией, Polly v8: timeout → retry → circuit breaker → attempt timeout.

Живой прогон: 7 заказов — 6 дошли до Completed сквозь случайные 500-е (ретраи видны в логах), лимитный корректно отменён компенсацией. 402 не ретраится — бизнес-отказ ≠ transient.

Study-guide README
Запуск: docker compose up -d --wait → три терминала: Bank, Payments, Orders (команды в README)

5 · GrpcLab

готов
Раздел 10 · API: REST и gRPC

Один метод «получить долг» в двух транспортах на одном сервере: gRPC (protobuf/HTTP2) и REST (JSON/HTTP1.1). Клиент замеряет p50/p95/p99, показывает server streaming и deadline (отменяет работу на обеих сторонах). В .proto — комментарии о правилах эволюции контракта.

Сюрприз: на localhost с крошечным payload REST быстрее gRPC (0.18 vs 0.28 мс p50) — и это корректный результат для такого профиля. Плюс история про нечестный бенчмарк: interceptor логировал каждый вызов и портил замер.

Study-guide README
Запуск: два терминала — Server, затем Client · без инфраструктуры

6 · RedisLab

готов
Раздел 11 · Кэширование и Redis

Redis + «медленная БД» со счётчиком обращений. Три сценария: cache-aside (225 мс → 1 мс), cache stampede — 50 конкурентных промахов тремя способами, инвалидация через DEL при записи. Распределённый лок собран по-честному: SET NX PX + токен владельца + Lua compare-and-delete.

Stampede цифрами: наивный cache-aside — 50 походов в БД, single-flight (SemaphoreSlim) — 1, распределённый лок — 1 (и работает между инстансами).

Study-guide README
Запуск: docker compose up -d → dotnet run --project src/RedisLab.App -c Release

7 · TestingLab

готов
Раздел 13 · Тестирование

Интеграционные тесты на Testcontainers: реальный Postgres поднимается из теста сам (и сам умирает через ryuk). Один контейнер на класс + TRUNCATE между тестами. Каждый из 4 тестов — про поведение, которое in-memory провайдер не поймает: unique-индекс, сырой SQL, xmin-конкуренция.

Главный тезис: с in-memory провайдером тест на дубликат договора «зелёный», а прод падает. Зелёный тест, красный прод — худший из исходов.

Study-guide README
Запуск: dotnet test · нужен только запущенный Docker

8 · CleanLab

готов
Раздел 7 · Архитектура: DDD, Clean, CQRS

Модуль «долги» на четырёх слоях: Domain (агрегат с инвариантами, value object Money, доменные события) ← Application (CQRS через MediatR, порты) ← Infrastructure ← Api. Все стрелки зависимостей — внутрь. Плюс честная секция «где это оверинжиниринг».

Выигрыш виден в тестах: домен тестируется голым new, application — без моков БД. Мысленный эксперимент «замени in-memory на EF» затрагивает один файл.

Study-guide README
Запуск: dotnet test · API на :5400 · без инфраструктуры

9 · K8sLab

готов
Раздел 12 · Контейнеры, Kubernetes, CI/CD

AspNetLab в локальном кластере kind: multi-stage Dockerfile, Deployment с probes и limits, ConfigMap → .NET-конфиг, Service, HPA. Затем те же манифесты — Helm-чартом, и референсный GitHub Actions пайплайн (build → test → образ по SHA → helm --wait).

10 · PerfLab

готов
Раздел 14 · Производительность и highload

Две части: BenchmarkDotNet — парсинг через Span (0 байт аллокаций против 168 КБ) и отчёт через StringBuilder (×20 быстрее конкатенации); NBomber — p50/p95/p99 от RPS для пары эндпоинтов, различающихся одной строкой: await Task.Delay против Thread.Sleep.

Thread pool starvation вживую: async плоский на любом RPS, blocking на 400 RPS — p99 ×7 (476 мс). Симптом из прода: «латентность растёт, а CPU спит».

Study-guide README
Запуск: бенчмарки -- --job short · нагрузка dotnet run (~1 мин, самодостаточна)

11 · ObservabilityLab

готов
Раздел 15 · Наблюдаемость

OpenTelemetry поверх OrderFlow (код живёт там): трейсы через HTTP, Outbox и Kafka в Jaeger, метрики в Prometheus. Kafka-propagation руками (inject/extract заголовков), producer/consumer-спаны, Npgsql-спаны одной строкой.

Один трейс саги — 21 спан через три сервиса, в нём видны все 4 ретрая Polly. Пойманный нюанс: трейс рвался на outbox-таблице — починено сохранением traceparent в строке события.

Study-guide README
Запуск: стек OrderFlow · Jaeger UI на :16686, Prometheus на :9090

Раздел 16 · System design

Пять полных разборов по шаблону «требования → нагрузка → схема → детализация → bottleneck»: платёжный шлюз (твой домен, начинай собес с него), URL shortener, нотификации, вебхуки, лента. Каждый опирается на паттерны из кодовых проектов — Outbox, идемпотентность, cache-aside, fan-out.

13 · SolidMap

чтение
Раздел 6 · ООП, SOLID, паттерны

Карта теории на живых примерах из твоих же проектов: каждая буква SOLID и девять паттернов GoF (Strategy, Template Method, Decorator, Observer, Proxy…) — со ссылкой на конкретный файл, где это реализовано. Плюс архитектурные паттерны (Outbox, Saga, CB) с отсылками к их study-guide'ам.

Главный приём: на «расскажи про SOLID» отвечать не определениями, а парой «принцип → своя история из кода». Одна конкретика дороже пяти определений.
