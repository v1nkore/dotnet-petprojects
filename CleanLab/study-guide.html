2Ключевые идеи

Правило зависимостей. Все стрелки — внутрь: Api → Infrastructure → Application → Domain. Внутренний слой не знает об внешнем. Достигается инверсией: Application объявляет IDebtRepository, Infrastructure реализует. Смена БД/фреймворка/транспорта не трогает бизнес-логику.

Value object vs entity. Money — без идентичности, равна по значению, иммутабельна (readonly record struct). Debt — есть Id, живёт во времени, меняется. VO поднимает инварианты на уровень типа: «сложить рубли с долларами» не компилируется в баг, а бросает доменную ошибку в одном месте.

Агрегат — граница согласованности. Все инварианты долга (не переплатить, не платить в закрытый) защищены методами агрегата; снаружи некорректное состояние недостижимо. Правило: транзакция меняет ОДИН агрегат; между агрегатами — eventual consistency через события (мост к OrderFlow).

Доменные события. Агрегат фиксирует факт (DebtClosed) и копит в списке; application-слой публикует ПОСЛЕ сохранения (подписчики не должны реагировать на несохранённое). Побочные эффекты (SMS) — в подписчиках, а не в агрегате. В проде публикация — через Outbox.

CQRS без карго-культа. Команды идут через агрегат (инварианты!), запросы — напрямую в DTO (читать через агрегат — таскать инварианты туда, где они не нужны). Здесь одно in-memory хранилище: CQRS — это разделение МОДЕЛЕЙ, а не обязательно баз. Отдельная read-база — это уже CQRS + репликация, когда чтение реально не влезает.

Где оверинжиниринг (любимый вопрос на собесе): CRUD без инвариантов, прототипы, маленькие сервисы с одной моделью — там эти слои чистый налог. Признаки, что пора: настоящие бизнес-инварианты, несколько транспортов (HTTP+gRPC+консьюмер), долгая жизнь кода. MediatR тоже опционален — те же руки можно сделать обычными сервисами; его ценность — pipeline behaviors и единообразие.
