# GrpcLab — gRPC против REST, deadline и interceptor

Pet-проект №7 из плана. Закрывает раздел 10 из «200 ответов»: **API: REST и gRPC**.

## Предметная область

Один и тот же метод «получить долг» на одном сервере в двух транспортах:
gRPC (protobuf/HTTP2, порт 5501) и REST (JSON/HTTP1.1, порт 5502). Клиент замеряет
латентность обоих, демонстрирует server streaming и deadline.

## Как запустить

```bash
dotnet run --project src/GrpcLab.Server -c Release    # терминал 1
dotnet run --project src/GrpcLab.Client -c Release    # терминал 2
```

## Результат с этой машины (2000 последовательных unary-вызовов, localhost)

| | p50 | p95 | p99 |
|---|---:|---:|---:|
| gRPC (protobuf/HTTP2) | 0.283 мс | 0.373 мс | 0.507 мс |
| REST (JSON/HTTP1.1) | **0.184 мс** | **0.265 мс** | **0.356 мс** |

**Сюрприз: REST быстрее.** И это правильный результат для данного профиля: крошечное
сообщение, loopback, последовательные вызовы. Фиксированная цена HTTP/2-фрейминга
перевешивает экономию protobuf, а мультиплексирование в последовательном цикле не даёт
ничего. gRPC выигрывает при больших payload'ах, реальной сети, параллельных вызовах
и стриминге. «gRPC быстрее REST» — не аксиома, а функция от профиля нагрузки —
готовый ответ уровня senior вместо заученной мантры.

## Что где смотреть

| Тема | Где | Суть |
|---|---|---|
| Контракт и эволюция | `Server/Protos/debts.proto` | Один .proto генерирует обе стороны; правила: номера полей не переиспользовать, ноль enum'а — UNSPECIFIED, деньги — int64 в копейках (в proto3 нет decimal) |
| Unary + streaming | `Server/Services/DebtsService.cs` | Server streaming для выгрузок/лент; `context.CancellationToken` отменяется при разрыве и deadline |
| Ошибки | там же | `RpcException` + `StatusCode` (InvalidArgument ≈ 400, Unavailable ≈ 503-ретраибельный) вместо HTTP-статусов |
| Deadline | `Client/Program.cs` §4 | Едет с запросом (заголовок `grpc-timeout`), отменяет работу на ОБЕИХ сторонах — в отличие от клиентского timeout |
| Interceptor | `Server/Services/LoggingInterceptor.cs` | Аналог middleware для gRPC; логирует только медленные вызовы — лог каждого вызова сам портит латентность (испортил первый замер этого бенчмарка) |
| Два протокола на Kestrel | `Server/Program.cs` | gRPC требует HTTP/2 (h2c без TLS локально); REST рядом на HTTP/1.1 |

## Пойманный нюанс

Первый замер показал gRPC ещё хуже — оказалось, interceptor логировал каждый из 2000
вызовов, а REST-эндпоинт не логировал ничего. Нечестный бенчмарк хуже отсутствия
бенчмарка. После исправления разрыв сократился, но вывод не изменился — и это тоже урок.
