1Предметная область и общая картина

клиент ──POST /orders──▶ [Orders :5300] ──OrderCreated──▶ [Payments] ──POST /charge──▶ [Bank :5301 💥]
                             ▲                                 │
                             └────── PaymentSucceeded / PaymentFailed ──────┘

  Pending ──оплата прошла──▶ Completed
  Pending ──оплата НЕ прошла──▶ Cancelled   ← компенсирующая транзакция

Это минимальная честная распределённая система: две базы, брокер, ненадёжная сеть. Одной ACID-транзакцией «создать заказ и списать деньги» здесь не сделать — и весь проект о том, какими паттернами это компенсируется: Outbox, идемпотентность, сага, ретраи с circuit breaker'ом.

«Банк» намеренно ужасен (35% пятисоток, зависания, лимиты) — это модель реального внешнего API. Всё, что делает Payments, продиктовано его ненадёжностью.

3Outbox: обе половины

Проблема (dual write): сохранить заказ в свою БД и отправить событие в Kafka — два разных хранилища, общей транзакции нет. Порядок «сначала БД, потом Produce» теряет событие при падении между ними; «сначала Produce» — создаёт событие о заказе, которого нет.

Решение: событие пишется в таблицу OutboxMessage той же транзакцией, что бизнес-данные (в EF — просто два Add и один SaveChanges). Отдельный воркер (OutboxPublisher<TContext>) опрашивает таблицу, публикует, помечает PublishedAtUtc.

Цена: если воркер упал между ProduceAsync и SaveChanges — после рестарта сообщение уйдёт повторно. Outbox даёт at-least-once, и это не баг, а осознанный размен: потерять нельзя, задублировать можно. Дубликаты лечатся на приёмнике — раздел 4.

Детали в коде: частичный индекс WHERE "PublishedAtUtc" IS NULL (воркер сканирует только хвост), SaveChanges на каждое сообщение (окно повтора — одно сообщение, не батч), продюсер с Acks.All + EnableIdempotence (брокер дедуплицирует ретраи самого продюсера).

4Идемпотентность: почему без неё нельзя

At-least-once означает: дубликаты — вопрос «когда», а не «если». Для платежей повторная обработка = повторное списание. Схема защиты (см. OrderEventsConsumer):

1. пришло сообщение (MessageId в конверте)
2. SELECT из ProcessedMessage: уже есть? → молча выходим
3. бизнес-эффект (списание)          ┐
4. INSERT в ProcessedMessage         ├─ ОДНА транзакция БД
5. INSERT исхода в свой Outbox       ┘
6. commit offset'а Kafka

Упали после шага 5, но до 6? Kafka передоставит, шаг 2 отсечёт дубль. Упали после 3, но до 4? Транзакция не закоммичена — эффекта нет, обработаем заново с нуля. Ключевой инвариант: эффект и отметка «обработано» атомарны.

5Saga-хореография и компенсация

Сага — последовательность локальных транзакций, где каждая публикует событие, запускающее следующую. Здесь — хореография: нет центрального мозга, сервисы просто реагируют на события друг друга. Альтернатива — оркестрация: отдельный компонент (state machine, например MassTransit Saga) явно командует шагами.

ХореографияОркестрация
Связанностьниже — сервисы знают только событияоркестратор знает всех
Видимость процессаразмазана по логам сервисовв одном месте, легко дебажить
Когда2–3 шага, простые компенсациидлинные процессы, ветвления, таймауты шагов

Компенсация ≠ rollback. Откатить чужую БД нельзя — можно только применить новое действие, гасящее бизнес-эффект: заказ не «исчезает», а переходит в Cancelled. Отсюда важное следствие — eventual consistency: между OrderCreated и Cancelled наблюдатель видит Pending-заказ, который «как будто есть». Система обязана уметь жить в этом окне.

6Kafka-механика: партиции, offset'ы, консьюмер

  • Партиция — единица порядка и параллелизма. Порядок гарантирован только внутри партиции. Ключ сообщения хэшируется в номер партиции → ключ = OrderId означает «события одного заказа строго по порядку». Без ключа PaymentSucceeded мог бы обогнать OrderCreated.
  • Consumer group: партиции топика делятся между консьюмерами группы; два сервиса с разными group.id читают один топик независимо (у каждой группы свои offset'ы). Rebalance — перераздача партиций при входе/выходе участника; Close() в коде — мгновенный выход из группы вместо ожидания session timeout.
  • Offset — курсор группы в партиции. EnableAutoCommit=false + ручной Commit после обработки = at-least-once. Auto-commit по таймеру может закоммитить то, что ещё не обработано = at-most-once (потеря при падении).
  • Consume() — блокирующий вызов, поэтому цикл живёт на выделенном потоке (TaskCreationOptions.LongRunning), а не отбирает поток у пула. Там же — редкий легальный случай sync-over-async: нет SynchronizationContext → deadlock невозможен.
Kafka vs RabbitMQ (краткий ответ для собеса): Kafka — распределённый лог: сообщения не удаляются при чтении (retention), группы читают независимо, возможен replay с любого offset'а; сила — throughput, порядок, event sourcing, несколько независимых потребителей одного потока. RabbitMQ — умный брокер с очередями: сложная маршрутизация (exchanges), per-message ack, TTL/приоритеты; сила — классические задачи «очередь работ». Здесь нужна история событий и независимые потребители → Kafka.

7Polly: анатомия pipeline

builder.Services.AddHttpClient<BankClient>(...)
    .AddResilienceHandler("bank", pipeline => {
        pipeline.AddTimeout(10s);          // 1. потолок ВСЕЙ операции
        pipeline.AddRetry(4, exp, jitter); // 2. ретраи transient-ошибок
        pipeline.AddCircuitBreaker(...);   // 3. перестать добивать лежащего
        pipeline.AddTimeout(1s);           // 4. потолок ОДНОЙ попытки
    });

Стратегии — луковица, выполняются сверху вниз; порядок = семантика. Attempt-timeout внутри retry — иначе одна зависшая попытка съест весь бюджет. Circuit breaker между ними: считает результаты попыток и при 90% отказов за окно размыкается — быстрые отказы вместо мучения банка (и своих потоков).

  • Jitter обязателен: без него сто инстансов ретраят синхронно — retry storm добивает восстанавливающийся сервис.
  • Что ретраить: 5xx, таймауты, сетевые ошибки. Что нельзя: 4xx-бизнес-отказы (402 «лимит» детерминирован — в коде это отдельная ветка BankChargeResult.Declined → сразу компенсация).
  • Реальный лог прогона — в README: два 500 → ретраи → OK; зависание → attempt timeout → OK; 402 → без ретраев → Cancelled.
Тонкость: устойчивость навешана на HttpClient снаружи (AddResilienceHandler = DelegatingHandler), сам BankClient о ней не знает. Retry на боевом POST безопасен только если внешняя система идемпотентна (реальный банк принимал бы idempotency key) — у нас «банк» stateless, поэтому честно.

8Мостик к вопросам собеса

Kafka: партиции/offset/consumer group; гарантии доставки (at-most/at-least/exactly-once и почему «exactly-once» — это на самом деле идемпотентность приёмника); ключ партиционирования и порядок; rebalance; auto-commit vs ручной; Kafka vs RabbitMQ.
Микросервисы: dual write и Outbox; идемпотентный консьюмер (инвариант «эффект + отметка атомарны»); saga: хореография vs оркестрация; компенсация vs rollback; eventual consistency и что она значит для UI/бизнеса.
Устойчивость: retry/backoff/jitter; circuit breaker (зачем, параметры, half-open); timeout total vs per-attempt; порядок стратегий; что нельзя ретраить; retry storm.
Бонус: почему consumer в выделенном потоке; когда sync-over-async легален; graceful shutdown консьюмера (Close vs Dispose).
