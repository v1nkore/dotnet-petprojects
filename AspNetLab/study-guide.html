1Предметная область

Сервис нотификаций: POST /notifications принимает сообщение, кладёт во внутреннюю очередь и мгновенно отвечает 202 Accepted со ссылкой на статус; фоновый воркер отправляет. Это классический паттерн «принял — подтвердил — обработал позже»: HTTP-запрос не должен ждать медленный SMTP/SMS-шлюз.

Домен нарочно крошечный. Смысл проекта — не нотификации, а то, из чего сделан любой сервис на ASP.NET Core: конвейер обработки запроса, контейнер зависимостей и фоновая работа. На собесе спрашивают именно эти три вещи.

3Pipeline: как запрос проходит сквозь приложение

запрос
  → ExceptionHandler      (ловит всё, что вылетит ниже → ProblemDetails 500)
  → StatusCodePages       (голым 4xx/5xx без тела дописывает ProblemDetails)
  → CorrelationId         (заголовок + log scope — до всех логов ниже)
  → RequestTiming         (замер: всё, что ниже, попадает в тайминг)
  → Routing → [фильтры MVC: биндинг → валидация → action-фильтры] → Action
ответ ← тем же путём обратно (у middleware есть «до next» и «после next»)

Порядок — это семантика. ExceptionHandler первым, потому что ловит только то, что зарегистрировано после него. CorrelationId раньше таймингов, чтобы лог тайминга уже был в scope с id. Поменяй местами — и логи потеряют correlation id, а падение middleware выше обработчика уронит соединение без ProblemDetails.

Middleware vs фильтр — когда что: middleware видит сырой HTTP и работает для ВСЕХ запросов (авторизация, логирование, сжатие). Фильтр живёт внутри MVC: после маршрутизации и биндинга, видит типизированные аргументы action'а, знает, какой контроллер вызывается. Валидация «Recipient соответствует Channel» требует распарсенную модель → это фильтр. Correlation id нужен всем, включая 404 до MVC → это middleware.

ProblemDetails (RFC 9457) — стандартный формат тела ошибки. AddProblemDetails + CustomizeProblemDetails дописывает correlationId в каждую ошибку: саппорт по нему находит логи конкретного запроса. Валидационные 400 от [ApiController] — это тоже ProblemDetails (ValidationProblemDetails).

4DI lifetimes и captive dependency

LifetimeЖивётТипичное
Singletonвсё приложениекэш, очередь, HttpClient-фабрика, конфиг
Scopedодин HTTP-запрос (или созданный вручную scope)DbContext, UnitOfWork, текущий пользователь
Transientкаждый Resolve — новыйлёгкие stateless-хелперы

Captive dependency — singleton, получивший scoped в конструктор. Scoped-инстанс «пленён» навсегда: для DbContext это один контекст на все запросы сразу — гонки (DbContext не потокобезопасен), распухший change tracker, ObjectDisposedException в случайных местах. Худшее — это не падает сразу, а стреляет под нагрузкой.

// Program.cs — поэтому включено ВСЕГДА, не только в Development:
builder.Host.UseDefaultServiceProvider(o => {
    o.ValidateScopes = true;   // ловит resolve scoped вне scope в рантайме
    o.ValidateOnBuild = true;  // ловит captive dependency прямо на Build()
});

Запусти dotnet run --project src/AspNetLab.Api -- --BreakDi=true — приложение упадёт на старте с текстом Cannot consume scoped service 'INotificationSender' from singleton 'CaptiveTrap'. Тест CaptiveDependencyTests проверяет ровно это.

Нюанс, который любят на собесах: по умолчанию обе проверки включены только в Development. В проде контейнер молча отдаст singleton'у scoped-инстанс — и никто не узнает до первого инцидента. Включение навсегда стоит копейки: проверка выполняется один раз на старте.

5BackgroundService и очередь

Bounded Channel вместо самодельной очереди

System.Threading.Channels — стандартный producer/consumer: вместо lock + Queue + Monitor.Pulse. Ключевой выбор — bounded (ёмкость 100, FullMode.Wait): при заполнении продюсер ждёт — это backpressure. Unbounded «никогда не тормозит» — пока не съест всю память при отстающем консьюмере.

Scope на единицу работы

У воркера-singleton нет HTTP-запроса, значит нет и request scope. Правильный паттерн — «scope на сообщение», зеркало того, что ASP.NET Core делает для каждого запроса:

await using var scope = scopeFactory.CreateAsyncScope();
var sender = scope.ServiceProvider.GetRequiredService<INotificationSender>();
await sender.SendAsync(item, stoppingToken);

Сравни с CaptiveTrap: это два ответа на один вопрос «как singleton'у пользоваться scoped-сервисом» — неправильный (в конструктор) и правильный (scope на работу).

Смертельный нюанс BackgroundService: с .NET 6 необработанное исключение из ExecuteAsync останавливает ВЕСЬ хост (BackgroundServiceExceptionBehavior.StopHost по умолчанию). Одно упавшее сообщение без try/catch внутри цикла = мёртвый сервис. Поэтому в цикле: catch на сообщение → лог → статус Failed → едем дальше; отдельный catch на OperationCanceledException для штатной остановки.

IHostedService vs BackgroundService: первый — интерфейс StartAsync/StopAsync, где сам следишь за задачей и токеном; второй — базовый класс, дающий готовый каркас с одним ExecuteAsync(stoppingToken). Для «вечного цикла» — всегда BackgroundService.

Options pattern

AddOptions<T>().BindConfiguration(...).ValidateDataAnnotations().ValidateOnStart() — секция конфига становится типизированным объектом, а кривое значение валит приложение на старте с внятным сообщением, а не при первом использовании в 3 часа ночи. Три интерфейса доступа: IOptions<T> (singleton, снимок на старте), IOptionsSnapshot<T> (scoped, перечитывается на запрос), IOptionsMonitor<T> (singleton + событие изменения).

6Два реальных бага из этого проекта

Баг №1 — [property: Required] на позиционном record. Атрибут ушёл на сгенерированное свойство, а MVC требует его на параметре конструктора — и бросил InvalidOperationException в рантайме при первом POST (оба эндпоинта отвечали 500). Правильно: [Required] string Recipient без target'а. Мораль: у позиционных record'ов атрибут по умолчанию применяется к параметру, и MVC валидирует именно параметры.

Баг №2 — настройки JSON как часть контракта. Включили JsonStringEnumConverter на сервере (enum'ы строками — читаемый контракт, не ломается при перестановке членов), и тест перестал парсить ответ: его GetFromJsonAsync использовал дефолтные настройки без конвертера. Клиент и сервер должны сериализовать по одним правилам — в реале это решают общим пакетом контрактов или генерацией клиента из OpenAPI.

7Мостик к вопросам собеса

Pipeline: порядок middleware и что сломается при перестановке; middleware vs фильтры; как работает short-circuit; зачем ProblemDetails; куда положить обработку ошибок.
DI: три lifetime'а и типичные ошибки; captive dependency — что это, чем опасна, как ловить (ValidateOnBuild/ValidateScopes) и как правильно (scope на работу); почему DbContext scoped, а HttpClient — через IHttpClientFactory.
Фоновая работа: IHostedService vs BackgroundService; исключение в ExecuteAsync гасит хост; graceful shutdown через stoppingToken; очередь между HTTP и воркером — Channels, bounded vs unbounded, backpressure.
Options: IOptions vs IOptionsSnapshot vs IOptionsMonitor; валидация конфига на старте.
Тестирование: WebApplicationFactory — интеграционный тест без сокетов; partial class Program для top-level statements.
