Платёжный шлюз (твой домен — начинай собес с него)

1. Требования

  • Ф: принять платёж (карта/СБП), провести через банк-эквайер, вернуть статус, вебхук мерчанту, возвраты.
  • НФ: деньги не теряются и не дублируются (главное!), доступность 99.95%, p99 ответа API < 500 мс (сам процессинг асинхронный), аудит каждого перехода статуса.

2. Нагрузка

  • 1 млн платежей/день ≈ 12 RPS средних, пик ×20 (распродажи) ≈ 250 RPS записи. Немного! Платёжки падают не от RPS, а от консистентности и внешних зависимостей.
  • Запись платежа ~2 КБ × 1 млн/день ≈ 2 ГБ/день, хранить годы (комплаенс) → партиционирование по месяцам.

3. Схема

мерчант → API GW → Payment API → Postgres (платёж + outbox, ОДНА транзакция)
                                     ↓ outbox-воркер
                                   Kafka: payment.commands
                                     ↓
                              Processor (идемпотентный консьюмер)
                                     ↓ HTTP + retry/CB/timeout + idempotency key
                                 Банк-эквайер 💥
                                     ↓ исход → outbox → Kafka: payment.events
                       ┌─────────────┴──────────────┐
                 Webhook-воркер (мерчанту)      Ledger/отчётность

Это OrderFlow один в один, с усилениями ниже.

4. Детализация

  • Идемпотентность на входе: мерчант шлёт Idempotency-Key; уникальный индекс по нему; повтор → тот же ответ, не второй платёж. (Как таблица ProcessedMessage, но на HTTP-краю.)
  • Статусы: Created → Processing → Succeeded/Failed/Refunded — только через state machine, каждый переход пишется в append-only лог (аудит).
  • Вызов банка: idempotency key банка = наш payment id; retry только на timeout/5xx; после N неудач — статус Unknown + фоновая сверка статуса у банка (нельзя ретраить вслепую то, что могло пройти!).
  • Деньги: decimal в minor units; ledger двойной записью (каждая операция — дебет+кредит, сумма проводок = 0) — сверка находит расхождения арифметикой.
  • Вебхуки мерчанту: отдельная очередь, ретраи с backoff, подпись HMAC (см. разбор 04).

5. Bottleneck и эволюция

  • Первым падает банк-эквайер → circuit breaker + очередь сглаживает пик + второй эквайер с маршрутизацией.
  • Postgres на записи → партиционирование по времени; шардировать по merchant_id, когда один инстанс перестанет тянуть.
  • Сверка (reconciliation) с банком — ежедневный batch: расхождения будут всегда, вопрос — найдёшь ли ты их раньше клиента.

URL Shortener (классика — разминка любого собеса)

1. Требования

  • Ф: POST длинный → короткий код; GET /{code} → 301 редирект; TTL/кастомные алиасы — опционально; клики — счётчик.
  • НФ: редирект p99 < 50 мс; чтение доминирует (100:1); доступность важнее консистентности счётчиков.

2. Нагрузка

  • 100 млн новых/мес ≈ 40 RPS записи; чтение ×100 ≈ 4000 RPS.
  • Запись: код (7 симв) + URL (~500 Б) × 100 млн/мес ≈ 50 ГБ/мес → за 5 лет ~3 ТБ: одна БД не смертельно, но шардирование заложить.
  • Кодов надо: base62, 7 символов = 62⁷ ≈ 3.5 трлн — хватит навсегда.

3. Схема

запись:  API → генератор кода → БД (code → url)
чтение:  GET /{code} → кэш (Redis) → hit 95%+ → 301
                          ↓ miss
                         БД → заполнить кэш → 301
клики:   fire-and-forget в Kafka → агрегатор → счётчики батчами

4. Детализация

  • Генерация кода — сердце разбора. Варианты:
  1. hash(url) — коллизии, одинаковый URL = одинаковый код (иногда нежелательно);
  2. случайный + проверка уникальности — лишний запрос к БД;
  3. счётчик → base62 — монотонный, предсказуемый (утечка объёмов бизнеса), зато без коллизий;
  4. прод-ответ: диапазоны id — координатор (БД/ZooKeeper) выдаёт сервису блок [1000000..1099999], сервис раздаёт из памяти без походов в БД. Кончился — взял следующий.
  • Кэш: cache-aside + TTL (RedisLab), 20% ключей дают 95% трафика. Stampede на вирусной ссылке → single-flight (уже реализовано в RedisLab).
  • Счётчики кликов — не UPDATE на каждый клик (умрёт от contention): событие в Kafka, агрегатор пишет батчами раз в N секунд. Eventual consistency здесь норм.
  • 404/удалённые: кэшировать и негативные ответы (короткий TTL) — иначе боты по несуществующим кодам пробивают кэш в БД.

5. Bottleneck и эволюция

  • Чтение — уже за кэшем; следующий предел — сама БД по объёму → шардирование по коду (первые символы или hash % N), код самодостаточен для роутинга.
  • Геораспределение: редирект-ноды + реплики кэша по регионам, запись — в один регион (запись редкая).

Сервис нотификаций (email / SMS / push)

1. Требования

  • Ф: единый API «отправь», 3 канала, шаблоны, статус доставки, отписки/предпочтения.
  • НФ: OTP-SMS — секунды (критично), маркетинг — можно часами; не слать дважды; уважать лимиты провайдеров; ночью маркетинг не слать.

2. Нагрузка

  • 10 млн/день ≈ 120 RPS средних; маркетинговая кампания — всплеск: 1 млн сообщений «одним нажатием». Вывод: очередь обязательна, API только принимает.

3. Схема

клиенты → Notification API → валидация, предпочтения, дедуп
                               ↓ (обогащение шаблоном)
              Kafka: notif.high (OTP) | notif.bulk (маркетинг)   ← два топика = приоритеты
                               ↓
          Sender-воркеры по каналам (email/sms/push), rate limiter НА ПРОВАЙДЕРА
                               ↓ retry/CB (Polly)
                        Провайдеры 💥 (SMTP, SMS-шлюз, FCM/APNS)
                               ↓ статусы/бонсы → Kafka → Status store

Это AspNetLab (принял → 202 → воркер), выросший до продовой топологии.

4. Детализация

  • Приоритеты = отдельные топики/очереди, не поле в сообщении: OTP не должен стоять за миллионом маркетинга. У каждого — свой пул воркеров.
  • Дедупликация: idempotency key от клиента + окно «то же сообщение тому же адресату за N минут» (Redis SETNX c TTL).
  • Rate limiting провайдера: token bucket на каждого (SMS-шлюз примет 100/с — не больше); при превышении сообщение ждёт в очереди, а не дропается.
  • Ретраи: transient (timeout, 5xx) — backoff+jitter; постоянные (невалидный номер, hard bounce) — сразу в Failed + событие для чистки базы адресов. Различать — как 402 vs 500 в OrderFlow.
  • Статусы: отправлено ≠ доставлено; доставка приходит асинхронно (webhook от провайдера — см. разбор 04).
  • Тихие часы/предпочтения — на этапе приёма (отложить до утра = scheduled-очередь), не в воркере.

5. Bottleneck и эволюция

  • Провайдеры — всегда первые: мульти-провайдер с failover и распределением (80/20).
  • Kafka-партиции по получателю — порядок сообщений одному юзеру.
  • Шаблонизация тяжёлая (рендер HTML) → отдельный пул, кэш скомпилированных шаблонов (TtlCache!).

Обработка вебхуков (входящих, от провайдеров/банков)

1. Требования

  • Ф: принять POST от N внешних систем (банки, PSP, SMS-шлюзы), проверить подлинность, обработать ровно один раз, отдать 200 быстро.
  • НФ: не потерять НИ ОДНОГО (это деньги/статусы); отвечать < 1 с (иначе отправитель ретраит и заваливает); переживать их всплески и свои деплои.

2. Нагрузка

  • Обычно скромная (тысячи/мин), но: отправитель при недоступности ретраит с нарастанием — после твоего 10-минутного даунтайма прилетит буря. Проектируем на ×50 от среднего.

3. Схема

провайдер → API GW → Webhook receiver: проверил подпись → сырое тело в БД/Kafka → 200 OK
                                              ↓ (всё! никакой бизнес-логики до ответа)
                                     Kafka: webhooks.raw
                                              ↓
                              Обработчики по типам (идемпотентные)
                                              ↓ не смогли N раз
                                            DLQ + алерт

Золотое правило: принять ≠ обработать. Ресивер только валидирует и складывает — обработка асинхронная.

4. Детализация

  • Подлинность: HMAC-подпись тела ключом из личного кабинета (сравнение constant-time!); плюс allowlist IP, если провайдер публикует. Timestamp в подписи против replay-атак (окно ±5 мин).
  • Идемпотентность: у событий провайдера есть id → таблица processed (как в OrderFlow); нет id → hash(тело+тип). Провайдеры ШЛЮТ ДУБЛИ, это норма протокола (at-least-once с их стороны).
  • Порядок не гарантирован: payment.succeeded может прийти раньше payment.created → обработчик либо переупорядочивает по timestamp, либо state machine терпит «события из будущего» (отложить в retry).
  • Сырое тело хранить как пришло (юридический след + можно переиграть обработку после фикса бага — replay из Kafka).
  • Свой даунтайм: очередь переживает деплой; отправитель получит 200 от живого инстанса за балансировщиком.

5. Bottleneck и эволюция

  • Обработчики отстают от бури ретраев → скейлить консьюмеров (партиции!), ресивер держит, т.к. тонкий.
  • Один «ядовитый» вебхук валит консьюмер в цикле → DLQ после N попыток + алерт (учебное упрощение OrderFlow здесь уже недопустимо).
  • Много провайдеров → нормализация в единый формат событий на границе, дальше — общий конвейер.

Лента (feed) — новости/активность

1. Требования

  • Ф: пользователь видит ленту постов тех, на кого подписан, новые сверху; постинг; пагинация.
  • НФ: чтение ленты p99 < 200 мс; чтения ≫ записи (1000:1); свежесть — секунды, не мгновенно (eventual OK).

2. Нагрузка

  • 10 млн DAU × 10 открытий ленты = 100 млн чтений/день ≈ 1200 RPS (пик ×5).
  • 1 млн постов/день ≈ 12 RPS записи. Асимметрия — главный факт дизайна.
  • Подписки: медиана 200, но есть аккаунты с 10 млн подписчиков — проблема знаменитостей.

3. Схема — два подхода (сравнение и есть ответ)

Fan-out on WRITE (push):  пост → воркер раскладывает id поста
                          в Redis-ленту КАЖДОГО подписчика (list/zset, обрезка до 800)
   чтение: ZRANGE своей ленты → hydrate постов из кэша → мгновенно
   минус: пост знаменитости = 10 млн записей

Fan-out on READ (pull):   чтение → взять список подписок → последние посты каждого → merge
   минус: чтение дорогое, а чтений в 1000 раз больше

Прод-ответ — ГИБРИД: обычные авторы → push; знаменитости → pull,
их посты подмешиваются при чтении. Порог — по числу подписчиков (~10–100k).

4. Детализация

  • Хранение ленты: Redis zset feed:{userId} → member = postId, score = timestamp; обрезка до N последних (полная история — в БД, лента — кэш горячего).
  • Пагинация — keyset, не offset: курсор = score последнего показанного (ZRANGEBYSCORE max=cursor limit N); offset на глубоких страницах деградирует (замерено в идеях EfCoreLab).
  • Hydration: лента хранит id, тела постов — отдельный кэш post:{id} (cache-aside) — пост правится в одном месте, не в миллионах лент.
  • Постинг: API → Kafka → fan-out воркеры (это просто OrderFlow-конвейер); партиционирование по авторам.
  • Горячий ключ: лента знаменитости при pull-чтении — реплики кэша + локальный L1 (TtlCache!) на несколько секунд.

5. Bottleneck и эволюция

  • Fan-out воркеры отстают в пиковый вечер → это ок (лента отстаёт на секунды), скейлятся партициями.
  • Redis по памяти: 10 млн юзеров × 800 id × ~30 Б ≈ 240 ГБ → кластер, шардирование по userId; ленты «спящих» юзеров не материализуем вовсе (lazy rebuild при возврате).
  • Ранжирование вместо хронологии → отдельный сервис скоринга при чтении: архитектура не меняется, меняется сортировка.
