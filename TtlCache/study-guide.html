1Предметная область

In-memory кэш — самый частый инфраструктурный компонент бэкенда: до всякого Redis почти в каждом сервисе есть кэш прямо в памяти процесса — конфиги, справочники, права, токены, результаты дорогих запросов. Проект решает две его фундаментальные проблемы:

1. Конкурентный доступ. Кэш дёргают десятки потоков одновременно. Обычный Dictionary при параллельной записи ломается непредсказуемо: порча внутренних бакетов, зависший в бесконечном цикле поток, IndexOutOfRangeException. Нужна синхронизация — и вариантов несколько, с сильно разной ценой (это и меряет бенчмарк).

2. Устаревание и память. Закэшированное протухает — пользователь сменил имя, а кэш отдаёт старое. И кэш без вытеснения = утечка памяти с отложенным OutOfMemoryException. Отсюда TTL + двухступенчатая уборка: ленивая при чтении и фоновая для того, что никто не читает.

Почему не просто взять готовое? В проде — взять (IMemoryCache, HybridCache в .NET 9+). Смысл проекта — понять, что у них внутри и почему они устроены именно так. На собесе спрашивают ровно это.

3Разбор ключевых решений

ConcurrentDictionary как хранилище

Чтение в нём вообще без блокировок — volatile-чтение ссылки на бакет. Запись — lock striping: словарь держит массив локов, пишущие потоки на разных ключах почти не сталкиваются. Профиль кэша — read-heavy, поэтому это оптимальный выбор (бенчмарк это подтверждает цифрами).

readonly record struct CacheEntry

Запись кэша — readonly record struct (TValue, long, TimeSpan). Три выгоды:

  • struct → лежит по значению прямо в словаре, ноль дополнительных объектов в куче на запись, меньше давление на GC (раздел 1: struct vs class);
  • readonly → иммутабельность, безопасно читать из многих потоков без синхронизации;
  • record → бесплатное равенство по значению, которое используется в атомарном TryRemove(KeyValuePair) ниже.

Монотонное время: GetTimestamp, а не DateTime.UtcNow

Храним не «момент смерти», а пару «когда положили + сколько жить», где «когда» — TimeProvider.GetTimestamp() (обёртка над Stopwatch). Системные часы могут прыгнуть (NTP-синхронизация, ручной перевод) — тогда кэш на DateTime.UtcNow разом «протухнет» весь или наоборот оживит мёртвое. Монотонные часы идут только вперёд.

TimeProvider — время как зависимость

Абстракция времени из .NET 8. В проде — TimeProvider.System, в тестах — FakeTimeProvider, у которого часы проматываются вручную. Это то же самое DI-мышление, что и с IClock, только теперь стандартное в BCL: и PeriodicTimer, и Task.Delay умеют работать от него.

Двухступенчатая инвалидация

Ленивая: TryGet, наткнувшись на протухшую запись, сам её удаляет. Дёшево, но не спасает от ключей, которые больше никто не читает. Активная: фоновый свипер раз в интервал проходит по словарю. Так же устроен Redis (lazy + active expiration) — готовый ответ на вопрос «как Redis удаляет протухшие ключи».

Тонкое место №1: ленивое удаление делается через TryRemove(KeyValuePair.Create(key, entry)), а не TryRemove(key). Между «увидел протухшую запись» и «удалил» другой поток мог записать в этот ключ свежие данные — перегрузка с парой атомарно удаляет «ключ + именно то значение» и не тронет чужую свежую запись. Классическая гонка check-then-act и её решение.

PeriodicTimer + Task.Run для свипера

Почему не System.Timers.Timer: тот стреляет по расписанию независимо от того, закончился ли предыдущий тик — очистки могут наложиться. PeriodicTimer.WaitForNextTickAsync() — pull-модель: следующий тик берётся только когда обработан предыдущий. Плюс он принимает TimeProvider → свипер тестируем без реального ожидания.

Тонкое место №2 (реальный баг из этого проекта): сначала таймер создавался внутри Task.Run, и тест флаковал: FakeTimeProvider.Advance() успевал выполниться до того, как таймер зарегистрировался в провайдере — тик не наступал никогда. Лечение: создавать таймер синхронно в конструкторе. Мораль: код «до первого await» и код «после» выполняются в разное время и, возможно, в разных потоках.

Interlocked для hits/misses

_hits++ — это три операции (прочитать, увеличить, записать), два потока теряют друг друга обновления. Interlocked.Increment — одна атомарная CPU-инструкция (lock xadd), без блокировок. Чтение — Interlocked.Read, потому что на 32-битных платформах чтение long не атомарно (torn read). Прямой ответ на «зачем Interlocked, если есть lock».

ConfigureAwait(false) в CleanupLoopAsync

Библиотечный код не должен возвращаться в захваченный контекст вызывающего — ему всё равно, на каком потоке продолжаться, а вызывающему это может стоить deadlock'а (классика: .Result на UI/старом ASP.NET). Правило: в библиотеках — везде ConfigureAwait(false), в приложениях на ASP.NET Core — не важно (контекста нет).

GetOrAdd и честность про stampede

При одновременном промахе двух потоков на одном ключе фабрика выполнится дважды, выигрывает последняя запись — ровно так же ведёт себя ConcurrentDictionary.GetOrAdd (фабрика вызывается вне лока!). Для дешёвых фабрик это норм. Защита дорогих — Lazy<T> внутри словаря или SemaphoreSlim на ключ — тема проекта №13 (Redis, cache stampede).

5Бенчмарк: что показывает и как читать

dotnet run --project benchmarks/TtlCache.Benchmarks -c Release -- --job short   # ~3 мин
dotnet run --project benchmarks/TtlCache.Benchmarks -c Release                  # полный, ~15 мин

Нагрузка: N потоков × 200 000 операций, 90% чтений / 10% записей по 1024 ключам. Реальные цифры с этой машины (i5-13500H, .NET 10, short-run):

Метод1 поток4 потока8 потоковContentions (8)
lock + Dictionary3.06 ms39.2 ms192.8 ms1571
ConcurrentDictionary0.81 ms1.79 ms3.03 ms~1
ReaderWriterLockSlim3.32 ms19.7 ms79.9 ms0
  • Сюрприз №1: ConcurrentDictionary быстрее даже на одном потоке (3.8×). Интуиция «без конкуренции lock бесплатен» не подтвердилась: лок-фри volatile-чтение дешевле даже незахваченного Monitor, который на каждой из 200 000 операций делает interlocked-вход и выход.
  • 8 потоков — разрыв 64×. У lock все потоки сериализуются в очередь на одном мониторе: 1571 contention — потоки реально паркуются в ядре ОС. У ConcurrentDictionary — меньше одного contention на весь прогон.
  • Сюрприз №2 — ReaderWriterLockSlim: «много читателей параллельно» звучит идеально для кэша, и он правда в 2.4× лучше lock под конкуренцией (contentions = 0). Но всё равно в 26 раз хуже ConcurrentDictionary: на операциях по ~15 нс цена входа/выхода read-lock'а больше самой работы. RWLS окупается на длинных чтениях (десятки микросекунд+). Готовая история для собеса: «мы предполагали X, замерили — оказалось Y».
  • Allocated ≈ 0 на операцию — доступ к словарю не аллоцирует; байты в таблице — это Task'и на запуск потоков.
Правила честного замера: только Release (в Debug JIT не оптимизирует); ноутбук в сеть; смотреть не только Mean, но и StdDev/Median; не верить разнице в единицы процентов на short-прогоне.

6Мостик к вопросам собеса

Формулировки, к которым проект даёт готовую историю «делал руками»:

Раздел 1 · runtime: struct vs class и где живут; что даёт readonly struct; boxing и как его избежать; поколения GC и почему короткоживущие записи кэша — это нагрузка на Gen0; зачем record.
Раздел 2 · многопоточность: чем ConcurrentDictionary лучше lock+Dictionary и когда наоборот; почему RWLS часто медленнее Monitor; Interlocked vs lock vs volatile; check-then-act гонки и атомарные операции «сравнить-и-удалить»; async state machine и код до/после первого await; ConfigureAwait(false); зачем CancellationToken в фоновых циклах; PeriodicTimer vs Timer.
Бонус — тестирование: «как тестировать код, завязанный на время?» — TimeProvider/FakeTimeProvider, тесты TTL за миллисекунды без флаков.
