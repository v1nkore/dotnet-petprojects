# TtlCache — потокобезопасный in-memory кэш с TTL

Pet-проект №1 из плана подготовки: **внутренности C#/.NET и многопоточность**.
Закрывает первые два раздела из «200 ответов»: *C# и .NET runtime* + *Async и многопоточность*.

## Предметная область

In-memory кэш — самый частый инфраструктурный компонент в бэкенде: до Redis'а почти
всегда стоит кэш прямо в памяти процесса (конфиги, справочники, токены, результаты
дорогих вычислений). Две его фундаментальные проблемы, которые решает этот проект:

1. **Конкурентный доступ.** Кэш читают и пишут десятки потоков одновременно —
   обычный `Dictionary` при этом ломается (порча внутреннего состояния, зависания,
   исключения). Нужна стратегия синхронизации — и их несколько, с разной ценой.
2. **Устаревание данных (TTL) и утечка памяти.** Данные протухают, а кэш без
   вытеснения — это утечка памяти с отложенным `OutOfMemoryException`. Нужны
   ленивая инвалидация при чтении + фоновая уборка того, что никто не читает.

## Ключевые решения (и почему так)

| Решение | Почему |
|---|---|
| `ConcurrentDictionary` как хранилище | Чтение lock-free, запись через lock-striping — под read-heavy профилем кэша быстрее любого «словарь + лок» (см. бенчмарк) |
| `readonly record struct CacheEntry` | Запись кэша лежит в словаре по значению: ноль лишних объектов в куче, меньше давление на GC |
| `TimeProvider` вместо `DateTime.UtcNow` | Время — инжектируемая зависимость: тесты на TTL проматывают часы за миллисекунды и не флакуют |
| `GetTimestamp()` вместо `GetUtcNow()` | Монотонные часы: перевод системного времени (NTP, ручной) не «протухнет» и не «оживит» записи |
| `PeriodicTimer` для свипера | Тики не накладываются друг на друга, если очистка затянулась; принимает `TimeProvider` |
| `TryRemove(KeyValuePair)` при ленивом удалении | Атомарно удаляет «ключ + именно это значение» — не снесёт запись, которую другой поток успел обновить |
| `Interlocked.Increment` для hits/misses | Атомарные счётчики без лока; `_hits++` из двух потоков теряет обновления |

## Результаты бенчмарка (short-run, i5-13500H, .NET 10)

Смешанная нагрузка 90% чтений / 10% записей, 200 000 операций на поток:

| Метод | 1 поток | 4 потока | 8 потоков | Lock contentions (8) |
|---|---:|---:|---:|---:|
| `lock` + Dictionary | 3.06 ms | 39.2 ms | 192.8 ms | 1571 |
| `ConcurrentDictionary` | **0.81 ms** | **1.79 ms** | **3.03 ms** | ~1 |
| `ReaderWriterLockSlim` | 3.32 ms | 19.7 ms | 79.9 ms | 0 |

Выводы:
- `ConcurrentDictionary` быстрее **даже на одном потоке** — лок-фри чтение дешевле даже незахваченного монитора;
- на 8 потоках разрыв с `lock` — **64×**: потоки сериализуются в очередь на одном мониторе (1571 contention);
- `ReaderWriterLockSlim` лучше `lock` под конкуренцией, но всё равно в ~26 раз хуже `ConcurrentDictionary`: на коротких операциях цена входа/выхода из read-lock съедает выигрыш от параллельного чтения.
